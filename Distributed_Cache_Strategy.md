# Distributed Cache Strategy for Extreme Scale

As the URL shortener scales to millions of requests per minute globally, a single Redis instance will become a bottleneck (both in terms of RAM limits and network throughput). To support this, we must transition to a **Distributed Cache Architecture**.

## 1. Redis Cluster Design

Instead of a standalone Redis server, we will deploy a **Redis Cluster**. 
Redis Cluster provides high availability and automatic partitioning across multiple Redis nodes.

### Architecture Topology
*   **Multiple Master Nodes:** The data is split across multiple Master nodes (e.g., 3 Masters). This multiplies the available RAM and spreads the CPU/Network load.
*   **Replication (High Availability):** Every Master has at least one Replica (Slave). If Master A crashes, Replica A is automatically promoted to Master within milliseconds without manual intervention.

### Connecting Spring Boot to a Redis Cluster
To connect our application to the cluster, we only need to update the `application.yml`:

```yaml
spring:
  data:
    redis:
      cluster:
        nodes:
          - 10.0.0.1:6379
          - 10.0.0.2:6379
          - 10.0.0.3:6379
```
Spring Data Redis (via Lettuce or Jedis driver) automatically handles cluster topology updates and redirects queries to the correct node.

## 2. Cache Sharding (Data Partitioning)

When using a Redis Cluster, the data must be distributed (sharded) evenly across all the Master nodes. Redis handles this automatically using **Hash Slots**.

### How Hash Slots Work
Redis Cluster contains exactly `16,384` hash slots. 
Every time the application tries to save or retrieve a URL mapping (e.g., `urls::abc1234`), Redis determines which node holds the data using this algorithm:

```text
HASH_SLOT = CRC16(KEY) mod 16384
```

*   `CRC16("urls::abc1234") % 16384` might equal `4500`.
*   If **Node 1** is assigned slots `0 - 5460`, the mapping `abc1234` is stored exactly on **Node 1**.
*   `CRC16("urls::xyz9999") % 16384` might equal `12000`.
*   If **Node 3** is assigned slots `10923 - 16383`, the mapping `xyz9999` is stored exactly on **Node 3**.

### Ensuring Even Distribution
Because our short codes (`abc1234`, `xyz9999`) generated by the Snowflake ID + Base62 algorithms are highly randomized and uniformly distributed, the `CRC16` hashing algorithm will naturally spread the URLs evenly across all 16,384 slots. 

This means if you have 3 Redis Masters, each will handle exactly ~33% of the URL mappings and ~33% of the redirect traffic, preventing "hot spots".

## 3. Advanced Strategy: Addressing the "Thundering Herd" Problem
Even with sharding, a single viral URL (e.g., a Super Bowl ad link) will hash to exactly **one** specific Redis node. If millions of people click that link simultaneously, that single Redis node could become overwhelmed (a hot key).

**Solution: Two-Tier Local Caching (Caffeine + Redis)**

If a single link becomes highly viral, we can introduce a micro-cache at the application level:
1.  **L1 Cache (In-Memory Caffeine):** Store the viral URL locally in the Spring Boot JVM memory with a very short TTL (e.g., 5 seconds). 
2.  **L2 Cache (Redis Cluster):** The standard distributed cache.

If 100,000 requests hit Spring Boot App Instance #1 for the viral link in a single second, the App fetches it from Redis *once*, saves it in its local JVM L1 cache for 5 seconds, and serves the remaining 99,999 requests directly from RAM without making a single network call to the Redis cluster.
